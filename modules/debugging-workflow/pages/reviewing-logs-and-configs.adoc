#  Reviewing logs and configs

= Reviewing Logs and Configurations for Orchestrator Workflows

In Red Hat Developer Hub (RHDH), Orchestrator serves as the engine for serverless workflows. When a workflow fails or exhibits unexpected behavior, the primary path to resolution is a systematic review of the backend logs and the associated workflow configurations.

== Understanding Orchestrator Logging
Orchestrator relies on the underlying RHDH backend infrastructure to capture execution traces. Logs are generated at the workflow execution level, capturing state transitions, task failures, and integration errors.

To effectively debug, you must distinguish between two primary sources of information:
1. **Workflow Execution Logs:** Generated by the workflow engine, providing insight into which specific step (task) failed within your workflow definition.
2. **Backend Infrastructure Logs:** Captured by the OpenShift pod running the RHDH backend, useful for identifying connectivity issues, authentication failures, or resource constraints.

== Reviewing Workflow Configurations
Before diving into logs, verify that your workflow configuration is syntactically and logically correct. Orchestrator uses JSON/YAML-based workflow definitions.

*   **Check Schema Validation:** Ensure your workflow definition complies with the expected schema. A common source of errors is incorrect mapping of input/output data between task nodes.
*   **Verify Endpoint Configurations:** Ensure that any external services (e.g., CI/CD triggers, database connectors) defined in your configuration are reachable from the cluster.
*   **Environment Variables:** Check the `app-config.yaml` or relevant Kubernetes Secrets/ConfigMaps to ensure that environment-specific variables used by the workflow (like API tokens or service URLs) are correctly injected.

== Hands-On Activity: Analyzing Workflow Execution Logs

In this activity, you will access the logs for a running or failed Orchestrator workflow via the OpenShift CLI.

### Prerequisites
*   Access to an OpenShift cluster with RHDH installed.
*   `oc` CLI tool configured with appropriate permissions.
*   A workflow currently deployed in the `rhdh` namespace.

### Steps

1.  **Identify the RHDH backend pod:**
    List the pods in your namespace to find the Orchestrator backend pod:
    [source,bash]
    ----
    oc get pods -n rhdh
    ----

2.  **Access real-time logs:**
    Stream the logs from the backend container to observe the workflow execution in real-time:
    [source,bash]
    ----
    oc logs -f deployment/rhdh-backstage -n rhdh
    ----

3.  **Filter logs for specific Workflow IDs:**
    To reduce noise, pipe the log output to `grep` to find entries related to your specific workflow execution ID:
    [source,bash]
    ----
    oc logs -f deployment/rhdh-backstage -n rhdh | grep "<WORKFLOW_INSTANCE_ID>"
    ----

4.  **Reviewing Configuration files in the UI:**
    *   Navigate to the **Orchestrator** tab in the Red Hat Developer Hub interface.
    *   Select the failed workflow instance.
    *   Examine the **Execution Details** pane. If a step failed, check the task input/output payload displayed in the UI to see if the JSON data matches the expected schema.

== Troubleshooting Tips
*   **Task Timeouts:** If you see "Task execution timed out" in the logs, check the workflow node configuration for `timeout` parameters. You may need to increase the threshold for long-running processes.
*   **Permission Denied:** If the logs indicate 401/403 errors, verify the service account tokens provided to the Orchestrator backend are still valid and possess the necessary RBAC roles to interact with the target resource.
*   **Loki Integration:** For persistent debugging, ensure your RHDH instance is configured to forward logs to an external aggregator like Loki, allowing you to search through historical execution logs rather than relying solely on pod restart retention.
