Configuring for partially disconnected OpenShift environments

Configuring for partially disconnected OpenShift environments

In partially disconnected environments, your OpenShift cluster has limited or restricted access to the public internet, requiring all container images and dependencies to be pulled from an internal, secure mirror registry. To successfully deploy Red Hat Developer Hub (RHDH) with the Orchestrator plugin, you must ensure that the environment is prepared to handle image redirections and that all necessary operators are available locally.

Prerequisites

Before beginning the installation, ensure the following requirements are met:

  • Your disconnected environment is configured with a local, private container registry.

  • You possess the necessary permissions to push OCI images to your internal registry.

  • The oc-mirror tool is installed, and its version matches your OpenShift Container Platform (OCP) cluster version.

  • You have identified your target registry URL and port (e.g., registry.example.com:5000).

Procedure: Configuring the Environment

Follow these steps to prepare your environment and deploy the Orchestrator plugin within a partially disconnected OpenShift cluster.

1. Mirroring Resources

Use the oc-mirror tool to sync the required images to your local registry.

  1. Create your ImageSetConfiguration file to include the RHDH and Orchestrator-related images.

  2. Execute the mirror command:

    oc-mirror --config mirror-config.yaml docker://<target-registry-url:port>

2. Redirecting Image Pulls

Once the images are mirrored, you must instruct the cluster to pull images from your local registry instead of the default external sources.

  1. Navigate to the working directory generated by the oc-mirror tool:

    cd <workspace folder>/working-dir/cluster-resources/

  2. Apply the generated ImageContentSourcePolicy and other cluster-wide resources:

    oc apply -f .

3. Installing Operators

With the registry configured, install the foundational operators required for Orchestrator functionality via the OpenShift OperatorHub.

  1. OpenShift Serverless Operator: Install this first to provide the underlying serverless capabilities.

  2. OpenShift Serverless Logic Operator: Install this to enable the workflow automation engine required by the Orchestrator plugin.

4. Deploying the Backstage Custom Resource (CR)

After the operators are running, you must configure the Backstage instance to recognize the Orchestrator plugin.

  1. Create a Backstage Custom Resource.

  2. Configure the CR to include the Orchestrator plugin dependencies. Refer to the specific plugin documentation for the exact YAML configuration required for your version of RHDH.

If your organization requires persistent data storage that is not managed by the default RHDH setup, you may optionally deploy a custom PostgreSQL database instance within your cluster at this stage. Ensure that the database service is reachable by the Backstage pod.

5. Finalizing Installation

Apply the final resource configuration:

oc apply -f backstage-cr.yaml

Once applied, monitor the deployment using the OpenShift web console or CLI:

oc get pods -n <your-rhdh-namespace>

Ensure the backstage pod and any associated orchestrator pods transition to the Running state. If the pods remain in ImagePullBackOff, verify that your ImageContentSourcePolicy was successfully applied and that your local registry contains the required images.
